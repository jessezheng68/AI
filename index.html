    社会影响研究

    AI 如何改变 Anthropic 的工作方式

    发布于 2025年12月3日

    AI 正在如何改变我们的工作方式？我们此前关于 AI 经济影响的研究从整体劳动力市场出发，涵盖了多种不同职业。但如果我们更深入地研究一些最早采用 AI 技术的人——也就是我们自己——会看到什么？

    将视角转向内部：在 2025 年 8 月，我们调查了 132 名 Anthropic 工程师和研究人员，进行了 53 次深入的质性访谈，并研究了内部 Claude Code 的使用数据，以了解 AI 的使用正在如何改变 Anthropic 的工作。我们发现，AI 的使用正在从根本上改变软件开发者的工作性质，同时带来希望与担忧。

    我们的研究揭示了一个正经历显著转型的职场：工程师完成的工作量大幅增加，变得更加“全栈”（能够胜任超出其通常专长的任务），学习和迭代速度加快，并开始处理此前被忽视的任务。这种广度的扩展也让人们思考其中的权衡——有人担心这可能意味着失去更深层次的技术能力，或变得不太能有效监督 Claude 的输出；而另一些人则欣然接受这个机会，能够更宏观、更高层次地思考问题。有人发现更多的 AI 协作意味着与同事的协作减少；有人也在思考自己是否最终会把自己自动化到失业。

    我们也意识到，在一家开发 AI 的公司里研究 AI 的影响意味着一种“特权视角”——我们的工程师能更早接触前沿工具，工作在相对稳定的领域，并且本身也在推动影响其他行业的 AI 转型。尽管如此，我们仍认为总体上研究并公开这些发现是有价值的，因为 Anthropic 内部工程师正在发生的变化，可能仍是更广泛社会转型的一个先行信号。我们的发现提示了一些可能值得各行业尽早关注的挑战与考量（但也请参见附录中的局限性部分）。数据收集时，Claude Sonnet 4 与 Claude Opus 4 是当时最强的模型，而能力仍在持续进步。

    更强的 AI 能带来生产力收益，但也提出了如何维持技术专长、保留有意义的协作，以及为一个可能需要全新学习、指导与职业发展方式的不确定未来做准备等问题。我们在下文的“展望未来”部分讨论了我们正在内部探索这些问题的一些初步步骤。我们也在近期关于 AI 相关经济政策想法的博客文章中探讨了潜在的政策回应。

    关键发现

    在本节中，我们简要总结来自调查、访谈以及 Claude Code 使用数据的发现。更详细的发现、方法与注意事项将在下文各部分展开。

    调查数据

    1. Anthropic 的工程师和研究人员最常用 Claude 来修复代码错误并学习代码库。调试和代码理解是最常见的用途（图 1）。
    2. 人们报告 Claude 的使用率和生产力收益都在提高。员工自述在 60% 的工作中使用 Claude，并获得 50% 的生产力提升，相比去年同期提高了 2–3 倍。这种生产力表现为：每个任务类别所花时间略少，但产出量显著增加（图 2）。
    3. Claude 辅助工作的 27% 属于如果没有它就不会去做的任务，例如扩大项目规模、制作“锦上添花”的工具（如交互式数据仪表板），以及如果手工完成将不划算的探索性工作。
    4. 大多数员工频繁使用 Claude，同时表示只能把 0–20% 的工作“完全委托”给它。Claude 是持续的协作伙伴，但使用它通常需要主动监督与验证，尤其是在高风险工作中——而不是把完全不需要验证的任务直接交给它。

    质性访谈

    1. 员工正在培养 AI 委托的直觉。工程师倾向于把任务委托给那些容易验证的工作（他们“相对容易快速嗅探/粗检正确性”）、低风险工作（例如“用完就扔的调试或研究代码”），或无聊的工作（“我对任务越兴奋，就越不太可能使用 Claude”）。许多人描述了一种信任递进：从简单任务开始，逐渐委托更复杂的工作——而他们目前仍把大多数设计或“品味”类任务留在人类手中，不过随着模型改进，这条边界正在被重新协商。
    2. 技能范围在扩展，但某些技能获得的练习在减少。Claude 让人们能够把技能扩展到软件工程的更多领域（“我现在可以非常胜任地做前端工作，或处理事务型数据库……以前我会害怕碰这些我不太擅长的东西”），但也有员工担心，用于编写和评审代码所需的更深层技能会出现某种意义上的退化——“当产出变得如此容易和快速时，真正花时间去学习一些东西变得越来越难。”
    3. 与编码技艺的关系在改变。一些工程师欣然接受 AI 辅助，专注于结果（“我以为我真的很享受写代码，而我想其实我只是享受写代码带来的成果”）；而另一些人则说，“确实有一些（写代码的）部分我会怀念。”
    4. 职场社交动态可能在改变。Claude 现在成为了过去会向同事询问的问题的第一站——因此有人报告导师与协作机会变少了。（“我喜欢与人合作，令人难过的是我现在‘需要’他们的地方变少了……更初级的人不像以前那样经常来问我问题了。”）
    5. 职业演化与不确定性。工程师报告他们正转向更高层次的工作，去管理 AI 系统，并报告显著的生产力提升。然而，这些变化也引发了关于软件工程作为职业的长期发展轨迹的疑问。有些人对未来怀有矛盾感：“短期内我感到乐观，但长期来看，我认为 AI 最终会做所有事情，让我和许多其他人变得无关紧要。”也有人强调真实的不确定性，只能说几年后自己的角色会是什么样“很难说”。

    Claude Code 使用趋势

    1. Claude 正在更自主地处理更复杂的任务。六个月前，Claude Code 在需要人类输入之前大约会自主完成 10 个动作。现在，它通常可以处理大约 20 个，需要更少的人类引导就能完成更复杂的工作流程（图 3）。工程师也越来越多地用 Claude 处理复杂任务，例如代码设计/规划（使用占比从 1% 增至 10%）与实现新功能（从 14% 增至 37%）（图 4）。
    2. Claude 修复了大量“痒点”（papercuts）。8.6% 的 Claude Code 任务涉及修复通常会被推迟、但能改善体验的小问题，比如为了可维护性重构代码（即“修复痒点”）。这些小修复可能累积成更大的生产力和效率提升。
    3. 每个人都变得更“全栈”了。不同团队以不同方式使用 Claude，通常用来增强他们的核心专长——例如安全团队用它分析不熟悉的代码，对齐与安全团队用它构建数据的前端可视化，等等（图 5）。

    调查数据

    我们调查了来自整个组织的 132 名 Anthropic 工程师和研究人员，了解他们日常具体如何使用 Claude。我们通过内部沟通渠道以及对覆盖研究与产品职能的多样化团队员工进行直接联系来分发问卷。我们在附录中加入了“局限性”部分，提供更多方法细节；同时我们也分享了调查问卷问题，便于他人评估我们的方法并将其改编用于自己的研究。

    人们用 Claude 做什么编码任务？

    我们请参与调查的工程师和研究人员评估他们使用 Claude 完成各种编码任务的频率，例如“调试”（使用 Claude 帮助修复代码中的错误）、“代码理解”（让 Claude 解释现有代码，帮助人类用户理解代码库）、“重构”（使用 Claude 帮助重组现有代码）、以及“数据科学”（例如让 Claude 分析数据集并绘制条形图）。

    下面展示了最常见的日常任务。大多数员工（55%）每天用 Claude 做调试。42% 每天用于代码理解，37% 每天用于实现新功能。更不常见的任务是高层设计/规划（可能因为这些任务人们倾向于保留在人类手中），以及数据科学和前端开发（可能因为这些任务总体更少见）。这与“Claude Code 使用趋势”部分报告的 Claude Code 使用数据分布大体一致。

    使用率与生产力

    员工自述，12 个月前，他们在 28% 的日常工作中使用 Claude，并从中获得 +20% 的生产力提升；而现在，他们在 59% 的工作中使用 Claude，并平均获得 +50% 的生产力提升。（当我们在工程组织中推广采用 Claude Code时，我们也观察到每位工程师每天合并的拉取请求（即成功纳入代码的变更）增加了 67%，这与上述结果大体相符。）这一年同比对比相当显著——暗示这两个指标在一年内都增加了 2 倍以上。使用率与生产力也强相关；在分布的极端端点，14% 的受访者通过使用 Claude 将生产力提升了 100% 以上——他们是我们内部的“重度用户”。

    需要补充说明的是（以及下文其它所有基于自述的生产力发现），生产力很难被精确衡量（更多局限性见附录）。AI 研究非营利组织 METR 的近期研究显示：有经验的开发者在高度熟悉的代码库中与 AI 协作时，会高估 AI 带来的生产力提升。尽管如此，METR 指出会导致生产力不如预期的因素（例如在大型、复杂环境中，或需要大量隐性知识/上下文的场景下 AI 表现更差）与我们的员工表示他们不会委托给 Claude 的任务类型高度吻合（见下文“AI 委托方式”）。我们在不同任务上自述的生产力提升，可能反映了员工正在形成更具策略性的 AI 委托技能——而这并未在 METR 的研究中得到体现。

    当我们询问员工：对于他们目前会使用 Claude 的任务类别，Claude 如何影响他们在该任务类别上的总体时间投入与产出规模时，一个有意思的生产力模式出现了。几乎所有任务类别都呈现出净时间减少，而产出规模的净增加更大：

    不过，当我们进一步深入原始数据时，会发现节省时间的回答在两端聚集——有些人反而会在 Claude 辅助的任务上花费显著更多时间。

    为什么会这样？人们一般解释说，他们需要做更多对 Claude 代码的调试与清理（例如“我把氛围编程玩到死胡同之后”），并且由于代码不是自己写的，在理解 Claude 的代码时要承担更多的认知负担。有些人提到，他们在“赋能”的意义上把更多时间花在任务上——有人说 Claude 帮助他们“在以前会立刻放弃的任务上坚持下去”；另一个人说它帮助他们做更彻底的测试，以及在新代码库中做更多学习与探索。总体来看，真正获得时间节省的人，可能是那些把快速可验证的任务切分给 Claude 的工程师；而那些花费更多时间的人，可能是在调试 AI 生成代码，或在 Claude 需要更多引导的领域工作。

    从我们的数据中也无法明确看出：自述节省的时间被重新投入到了哪里——是更多工程任务、非工程任务、与 Claude 的互动或对其输出的审查，还是工作之外的活动。我们的任务分类框架并不能捕捉工程师分配时间的所有方式。此外，节省时间的结果也可能反映自述中的感知偏差。要厘清这些效应，还需要进一步研究。

    产出规模的增加更直接、也更显著；几乎所有任务类别都出现更大的净增幅。考虑到人们在回答时针对的是任务类别（例如“调试”整体）而非单个任务，这一模式是合理的——也就是说，某人可以在“调试”这一类别上花费略少时间，却总体产生更多调试产出。生产力很难被直接测量，但这些自述数据表明，在 Anthropic，AI 主要通过更大的产出规模来实现生产力提升。

    Claude 让新的工作成为可能

    我们好奇的一点是：Claude 是否让人们能够完成性质上全新的工作，还是说这些由 Claude 辅助完成的工作，最终也会被员工完成（只是可能速度更慢）？

    员工估计，他们在 Claude 辅助下完成的工作中，有 27% 如果没有它就不会去做。工程师提到会用 AI 来扩展项目规模、完成一些“锦上添花”的内容（例如交互式数据仪表板）、做一些有用但繁琐的工作（如文档和测试），以及进行手工完成成本不划算的探索性工作。正如一位受访者解释的那样，他们现在可以修复更多过去会降低工作体验的“痒点”（papercuts），例如重构结构不佳的代码，或构建“能让另一项任务更快完成的小工具”。我们也在使用数据分析中寻找这一现象，发现 Claude Code 任务中有 8.6% 涉及“痒点修复”。

    另一位研究人员解释说，他们会同时运行多个版本的 Claude，让它们用不同方法探索同一个问题：

    人们往往把超强模型想成一个单一实例，就像得到一辆更快的车。但如果你有一百万匹马……就能测试一大堆不同想法……当你拥有额外的广度去探索时，它更令人兴奋，也更有创造性。

    正如我们在后续章节中将看到的，这些新工作往往涉及工程师去处理超出其核心专长的任务。

    有多少工作可以完全委托给 Claude？

    尽管工程师频繁使用 Claude，超过一半的人表示他们只能把 0–20% 的工作“完全委托”给 Claude。（需要说明的是，受访者对“完全委托”的理解可能不同：从完全不需要任何验证的任务，到足够可靠、只需要轻度监督的任务。）在解释原因时，工程师描述了他们会与 Claude 主动、迭代地协作，并验证其输出——尤其是在复杂任务或高风险领域，在这些领域代码质量标准至关重要。这表明工程师倾向于与 Claude 紧密协作并检查其工作，而不是在不验证的情况下直接交付任务，并且他们对什么算“完全委托”设定了很高的门槛。

    质性访谈

    这些调查发现揭示了显著的生产力提升与工作模式变化，但也引出了一个问题：工程师在日常工作中到底如何体验这些变化？为了理解这些指标背后的“人”的维度，我们对参与调查回复的 53 名 Anthropic 工程师与研究人员进行了深入访谈，以更深入地了解他们如何思考与感受职场中的这些变化。

    AI 委托方式

    工程师和研究人员正在发展多种策略，以便在工作流中更高效地利用 Claude。人们一般会把以下类型的任务委托出去：

    • 在用户的上下文之外且复杂度低：
      “我会把 Claude 用在那些我上下文不多、但整体复杂度也不高的事情上。”

      “我遇到的大多数基础设施问题都不难，Claude 就能处理……我对 Git 或 Linux 了解不多……Claude 能很好地弥补我在这些方面经验不足的问题。”
    • 容易验证：
      “对于那些验证成本相比生成成本不大的事情，它简直太厉害了。”
    • 定义清晰或相对独立：
      “如果项目的某个子组件与其余部分足够解耦，我就会让 Claude 先试试。”
    • 代码质量不关键：
      “如果是用完就扔的调试或研究代码，我就直接交给 Claude。如果概念上很难，或需要非常特定的调试注入方式，或是设计问题，我就自己做。”
    • 重复或无聊：
      “我对任务越兴奋，就越可能不使用 Claude。相反，如果我感到很抗拒……我常常发现，与其硬着头皮开做，不如先和 Claude 围绕这个任务开一段对话更容易开始。”在我们的调查中，平均而言，人们说 Claude 辅助的工作里有 44% 是他们自己并不享受去做的任务。
    • 提示比亲自执行更省时：
      “如果我预期一个任务花不到 10 分钟……我大概率不会费劲用 Claude。”

      “冷启动问题可能是现在最大的阻碍。所谓冷启动，是指：关于我团队代码库如何工作的很多内在信息，我是默认就知道的，但 Claude 默认并不知道……我可以花时间迭代出一个完美提示词[但]我大概率还是会直接自己去做。”

    员工在决定是否委托时提到的这些因素，与 METR 的一项外部研究中用来解释 AI 相关生产力放缓的因素（例如开发者对代码库非常熟悉、仓库规模巨大且复杂等）相似。我们访谈中对这些委托标准的趋同表明：恰当的任务选择是 AI 生产力收益的重要因素（未来的生产力研究应当仔细控制这一变量）。

    许多用户描述了他们使用 Claude 的一种演进过程：随着时间推移，逐步把更复杂的任务委托出去。“一开始我只是用 AI 工具问一些关于 Rust 编程语言的基础问题……最近，我已经用 Claude Code 做我所有的编码工作了。”

    一位工程师把这种信任递进比作采用 Google Maps 等其他技术：

    一开始我只在不认识的路线才用（Google Maps）……这就像我让 Claude 写我不会的 SQL，但不会让它写我会的 Python。后来我开始在大体熟悉、但最后一段路可能不太确定的路线也用 Google Maps……今天我几乎一直在用 Google Maps，甚至是日常通勤。如果它建议我换一条路走，我就照做，并信任它已经考虑了所有选项……我今天用 Claude Code 也是类似的方式。

    工程师对于要在自己熟悉的领域内还是领域外使用 Claude 存在分歧。有些人用它来处理“边缘”领域以节省实现时间；另一些人更偏好熟悉的领域，以便验证输出（“我使用 Claude 的方式是：我仍然能完全理解它在做什么”）。一位安全工程师强调了经验的重要性：当 Claude 提出一个方案时，这个方案“聪明得很危险，就像一个非常优秀的初级工程师可能会提出的那种”。也就是说，它的问题只有具备判断力与经验的用户才能识别出来。

    也有工程师两种任务都会用 Claude：要么以实验方式（“我基本上总是会让 Claude 先对任何编码问题打个头阵”），要么根据自己对该任务的熟悉程度来调整方法：

    我会把工具同时用于我专长范围内的事情（作为加速器：我知道该期待什么，也能有效引导代理），也用于略超出我专长范围的事情：我大体知道该期待什么，但 Claude 能帮我补上我在记忆或对具体定义熟悉度上的空白。

    如果是我特别熟悉的东西，我会更强势一些，直接告诉 Claude 需要去追查什么。如果是我不太确定的东西，我往往会让它当专家，给我选项和洞见，告诉我应该考虑和研究哪些点。

    人们也一致表示：他们不会在涉及高层或战略性思考的任务上使用 Claude，或在需要组织上下文或“品味”的设计决策上使用 Claude。一位工程师解释说：“我通常会把高层思考和设计留给自己。我会把我能从新功能开发到调试中委托出去的东西都委托出去。”这也反映在我们的调查数据中：在设计与规划任务上，生产力提升最小（图 2）。不过，许多人也把委托边界描述为“移动的目标”，随着模型提升而被不断重新协商（下文的 Claude Code 使用数据也显示：与六个月前相比，现在在编码设计/规划上的使用相对更多）。

    技能的变化

    新能力……

    调查中发现：Claude 辅助的工作里有 27% 如果没有 Claude 就不会发生——这反映了一个更广泛的模式：工程师在用 AI 去处理超出其核心专长的工作。许多员工报告，他们完成了以往不在自己专长范围内的工作——后端工程师搭 UI；研究人员做可视化。一位后端工程师描述了自己如何通过与 Claude 迭代构建一个复杂 UI：“它做得比我自己能做的好太多了。我肯定做不出来，更别说按时做出来了……[设计师们]会说‘等等，这是你做的？’我说‘不是，是 Claude 做的——我只是给它提示。’”

    工程师报告他们正在“变得更全栈……我现在可以非常胜任地做前端工作，或处理事务型数据库，或写 API 代码——以前我会害怕碰那些我不太擅长的东西。”这种能力扩展带来了更紧密的反馈回路与更快的学习：一位工程师说，以前需要“几周流程”才能完成的构建、约会、迭代，如今可以在同事在场、实时给反馈的情况下，变成“一两小时的工作会话”。

    总体而言，人们对自己获得的快速原型、并行化工作、减少低效劳动，以及总体提高雄心水平的能力感到兴奋。一位资深工程师告诉我们：“这些工具确实让初级工程师更高产，也更大胆，愿意接一些更有野心的项目。”也有人说，使用 Claude 所需的“启动能量”降低，让他们更容易战胜拖延，“大幅降低了我想开始解决一个问题所需的能量，所以我愿意去做更多额外的事情。”

    ……以及更少的亲身练习

    与此同时，有些人担心“当我委托更多时，技能在退化”，也担心在手动解决问题过程中发生的那些附带学习（或“连带学习”）会减少：

    如果你自己去调一个很难的问题，你会花时间读文档和代码——这些内容不一定直接有助于解决你手头的问题——但这整个过程会让你建立起系统如何工作的心智模型。现在这种事情少了很多，因为 Claude 可以直接把你带到问题上。

    我以前会把每个配置都探索一遍，去理解这个工具能做什么；但现在我依赖 AI 告诉我怎么用新工具，所以我缺少了专业性。以前和队友聊天时我能立刻回忆起很多东西；现在我得去问 AI。

    用 Claude 可能会跳过这样一个阶段：我先通过解决一个简单实例来学会如何做某件事；但之后遇到更复杂的实例时，我就会更挣扎。

    一位资深工程师说，如果自己更初级，他会更担心技能问题：

    我主要是在那些我知道答案应该是什么、或者答案应该长什么样的情况下使用 AI。我是通过“艰难地做软件工程”才培养出这种能力的……但如果我在职业早期，我会觉得需要非常刻意的努力，才能继续提升自己的能力，而不是盲目接受模型输出。

    编码技能退化之所以令人担忧，一个原因是“监督悖论”——如上所述，想要有效使用 Claude 需要监督，而监督 Claude 又需要那些可能会因过度使用 AI 而退化的编码技能。有人说：

    老实说，比起我的技能本身，我更担心监督与监管的问题……技能退化或无法发展，主要会影响我能否安全地用 AI 去做我关心的任务，而不是我是否还能独立完成这些任务。

    为此，一些工程师会刻意在不使用 AI 的情况下练习：“偶尔，即便我知道 Claude 能把某个问题做得很好，我也不会去问它。这能帮我保持锋利。”

    我们还需要这些亲身编码技能吗？

    软件工程或许正在走向更高层次的抽象——这在历史上也发生过。早期程序员离机器更近：手动管理内存、用汇编语言编程，甚至通过拨动物理开关来输入指令。随着时间推移，更高层、更易读的语言出现，自动处理复杂的底层操作。或许，特别是随着“氛围编程”的兴起，我们正在走向把英语当作编程语言。一位同事建议，有志成为工程师的人应该“擅长让 AI[写代码]，并专注于学习更高层次的概念与模式”。

    少数员工说，他们感到这种转变让他们能在更高层次思考——“关于最终产品和最终用户”，而不仅是代码。有个人把这种变化类比为以前在计算机科学课程里必须学链表：一种基础结构，而更高层的编程语言如今会自动处理。“我很庆幸我学过怎么做……[但]做那些底层操作在情感上并不重要。我更愿意关心代码让我能做什么。”另一位工程师做了类似类比，但指出抽象有代价——随着转向更高层语言，大多数工程师失去了对内存处理的深刻理解。

    继续在某个领域发展技能，可能带来对 Claude 更好的监督与更高效的工作（“我注意到当是我熟悉的东西时，我自己做往往更快”）。但工程师对这是否重要存在分歧。有些人仍然比较坦然：

    我不太担心技能侵蚀。AI 仍会让我仔细思考问题，也会帮助我学习新的方法。相反，能够更快地探索和测试想法，让我在一些领域的学习更快了。

    另一位则更务实：“我作为软件工程师的技能肯定在退化……但如果未来真的需要，它们也可以再回来，而且我现在就是不需要它们了！”还有人提到，他只丢掉了一些不那么重要的技能，比如做图表；而“关键的代码我仍然能写得很好”。

    也许最有意思的是，有位工程师质疑了前提：“所谓‘生疏了’的表述，隐含了一个假设：编码总有一天会回到 Claude 3.5 之前的样子。而我不认为它会。”

    软件工程的技艺与意义

    工程师们在是否怀念亲手写代码上分歧很大。有些人感到真实的失落——“对我来说，这是一个时代的终结——我写了 25 年程序，觉得自己在这项技能上很胜任，是我职业满足感的核心部分。”也有人担心自己不会喜欢新的工作形态：“整天对着 Claude 写提示词并不有趣，也不令人满足。真正更有趣、更满足的是放点音乐、进入心流，然后自己把东西实现出来。”

    有人直接谈到取舍并接受它：“确实有一些（写代码的）部分我会怀念——重构代码时进入那种禅定般的心流状态；但总体来说我现在生产力高太多了，我很乐意放弃这些。”

    也有人说，与 Claude 迭代更有趣，因为相比对人，他们可以更挑剔地给反馈。还有人更关注结果。一位工程师说：

    我本以为到了这个阶段我会感到害怕或无聊……但我并没有真正感觉到任何一种。相反，我很兴奋，因为我能做得更多。我以为我真的很享受写代码，而我想其实我只是享受写代码带来的成果。

    人们是拥抱 AI 辅助，还是哀悼亲手编码的流失，似乎取决于他们认为软件工程中哪些部分最有意义。

    职场社交动态的变化

    一个更突出的主题是：Claude 已经成为过去会问同事的问题的第一站。“我现在整体会问更多问题，但其中大概 80–90% 都问 Claude，”一位员工指出。这形成了一种过滤机制：Claude 处理日常、例行的询问，而同事则更多去处理超出 AI 能力的、更复杂、更具战略性或更依赖上下文的问题（“它把我对[团队]的依赖降低了 80%，[但]最后那 20% 非常关键，我还是会去跟他们聊”）。人们也会把 Claude 当作“弹想法”的对象，类似于与人类协作者的互动。

    大约一半的人报告团队协作模式没有变化。一位工程师说，他仍在与人开会、共享上下文、选择方向；他认为在不久的将来仍会有很多协作，但“你不再是在做你那种标准的专注工作，而是在和很多个 Claude 对话。”

    不过，也有人描述自己与同事的互动减少了（“我和 Claude 的协作远多于和任何同事的协作。”）。有些人喜欢社交摩擦的减少（“我不会因为占用同事时间而内疚”）。也有人抵触这种变化（“我其实不太喜欢大家的常见回应是‘你问过 Claude 了吗？’我很享受当面和人一起工作，而且非常珍视这一点”），或怀念以前的工作方式：“我喜欢和人一起工作，令人难过的是我现在‘需要’他们的地方变少了。”还有几个人指出，这会影响传统的导师机制，因为“Claude 可以给初级员工提供很多辅导”，从而替代了资深工程师。某位资深工程师说：

    更初级的人不像以前那样经常来问我问题了，这让人难过；不过他们的问题确实能更有效地得到解答，也学得更快。

    职业不确定性与适应

    许多工程师描述自己的角色正在从写代码转向管理 AI。工程师越来越把自己视为“AI 代理的管理者”——有人已经“总是至少同时开着几个 [Claude] 实例”。有人估计自己的工作已经“70% 以上变成代码评审/修改，而不是写全新的代码”；另一个人认为，未来角色的一部分是“为 1 个、5 个或 100 个 Claude 的工作负责”。

    从更长期看，职业不确定性十分普遍。工程师把这些变化视为行业更广泛转型的先兆，很多人说几年后自己的职业会是什么样“很难说”。有些人表达了短期乐观与长期不确定之间的矛盾：“短期内我感到乐观，但长期来看，我认为 AI 最终会做所有事情，让我和许多其他人变得无关紧要。”也有人把话说得更尖锐：“感觉就像我每天来上班，是来让自己失业的。”

    也有工程师更乐观。一位说：“我替初级开发者担心，但我也觉得初级开发者可能是最渴望新技术的一群。我总体上对这个职业的轨迹非常乐观。”他们认为，虽然确实存在经验不足的工程师发布有问题代码的风险，但随着更好的 AI 护栏、更内建的教育资源，以及人们从错误中自然学习的过程，行业会随着时间推移而适应。

    我们也问了人们如何想象自己未来的角色，以及是否有任何适应策略。有些人提到会进一步专精（“想要有意义地评审 AI 的工作，需要更久的时间，也需要更专业化”）；有些人预计未来会把重心放在更多人际与战略性工作上（“我们会花更多时间达成共识，让 AI 花更多时间做实现”）。也有人把 Claude 用于职业发展，让它就工作与领导力技能给自己反馈（“我能学习东西的速度，甚至是不完全学会也能有效工作的能力，都彻底改变了。我几乎感觉天花板被打碎了。”）。

    总体而言，许多人承认存在深度不确定性：“我对未来哪些具体技能会有用几乎没有信心。”一位团队负责人说：“没人知道会发生什么……重要的是要非常能适应。”

    Claude Code 使用趋势

    调查与访谈数据表明：更高的 Claude 使用率帮助人们更快工作并承担新的工作类型，但这也伴随着关于 AI 委托与技能发展的张力。尽管如此，基于自述的数据只讲述了故事的一部分。为补充这一点，我们也分析了 Anthropic 各团队的实际 Claude 使用数据。由于调查受访者报告 Claude Code 构成了他们使用的主要部分，我们使用隐私保护的分析工具，对 2025 年 2 月与 2025 年 8 月来自 Claude Code 的 20 万条内部对话记录进行了分析。

    用更少监督解决更难的问题

    在过去六个月里，Claude Code 的使用已转向更困难、更自主的编码任务（图 3）：

    • 员工正用 Claude Code 处理越来越复杂的任务。我们把每条对话记录的任务复杂度估计为 1–5 分：1 对应“基础编辑”，5 对应“需要人类专家数周/数月工作的专家级任务”。任务复杂度平均从 3.2 上升到 3.8。为说明分数差异：平均为 3.2 的任务包括“排查 Python 模块导入错误”，而平均为 3.8 的任务包括“实现并优化缓存系统”。
    • Claude Code 在每条记录中连续工具调用的最大次数增加了 116%。工具调用对应 Claude 使用外部工具采取的动作，例如编辑文件或运行命令。如今 Claude 能在不需要人类介入的情况下串联 21.2 次独立工具调用；而六个月前为 9.8 次。
    • 人类回合数减少了 33%。每条记录的平均人类回合数从 6.2 降到 4.1，表明与六个月前相比，现在完成同一任务所需的人类输入更少。

    这些使用数据印证了调查数据：工程师正把越来越复杂的工作委托给 Claude，而 Claude 需要的监督也更少。可以合理推测，这正在推动我们观察到的生产力提升。

    任务分布

    我们把 Claude Code 的对话记录归类为一种或多种编码任务类型，并研究不同任务类型的使用在过去六个月里如何演变：

    从使用数据估计得到的整体任务频率分布，与自述的任务频率分布大体一致。2025 年 2 月到 2025 年 8 月之间最显著的变化是：用 Claude 实现新功能的记录占比显著提高（14.3% → 36.9%），以及进行代码设计或规划的占比提高（1.0% → 9.9%）。Claude Code 任务相对分布的这种变化可能意味着 Claude 在这些更复杂任务上的能力增强了，但也可能反映团队在不同工作流中采用 Claude Code 的方式发生了变化，而非绝对工作量的增加（更多局限性见附录）。

    我们在调查中发现：工程师现在花更多时间做一些改善体验的小改进；与此一致的是，当前 Claude Code 任务中有 8.6% 被归类为“痒点修复”。这包括较大的任务，例如创建性能可视化工具、为了可维护性重构代码，也包括更小的任务，例如创建终端快捷方式。这可能有助于解释工程师自述的生产力提升（解决此前被忽视的体验改进，可能会在长期带来更高效率），也可能减少日常工作中的摩擦与挫败感。

    为了研究不同团队之间当前任务的差异，我们改进了分类方法：把 8 月的每条记录分配到一个主要编码任务上，并按内部团队拆分数据（y 轴）。堆叠条形图展示了每个团队的编码任务构成：

    “所有团队”一行展示整体分布，其中最常见的任务是实现新功能、调试与代码理解。这为团队间对比提供了基线。

    一些值得注意的团队特定模式：

    • 预训练团队（帮助训练 Claude）经常用 Claude Code 来实现新功能（54.6%），其中很大一部分是在运行额外实验。
    • 对齐与安全团队与后训练团队使用 Claude Code 做前端开发最多（7.5% 与 7.4%），常用于创建数据可视化。
    • 安全团队经常用 Claude Code 做代码理解（48.9%），具体是分析并理解代码库不同部分的安全影响。
    • 非技术员工经常用 Claude Code 做调试（51.5%），例如排查网络问题或 Git 操作，也会用于数据科学（12.7%）；Claude 似乎在弥合技术知识差距方面很有价值。

    这些团队特定模式展示了我们在调查与访谈中观察到的同样的能力扩展：让团队成员能做出一些原本没有时间或技能去做的新类型工作。例如，预训练团队运行了更多额外实验，非技术员工得以修复代码中的错误。而数据也表明：团队确实会用 Claude 做其核心任务（例如基础设施团队最常用 Claude Code 处理基础设施与 DevOps 工作），但 Claude 往往也会增强其核心任务（例如研究人员用 Claude 做前端开发，以更好地可视化数据）。这表明 Claude 正在让每个人在工作中都变得更“全栈”。

    展望未来

    Anthropic 员工在过去一年大幅增加了对 Claude 的使用，用它不仅来加速现有工作，也用来学习新的代码库、减少繁琐劳动、扩展到新的领域，以及处理此前被忽视的改进。随着 Claude 变得更加自主和能干，工程师们正在发现新的 AI 委托方式，同时也在思考未来需要什么技能。这些变化带来了明显的生产力和学习收益，同时也带来了对软件工程工作长期发展轨迹的真实不确定性。AI 会像过去的软件工程转型一样吗——从低级到高级编程语言，或从个人贡献者到管理者（正如几位工程师所建议的）？还是会走得更远？

    现在还为时尚早——Anthropic 内部有许多早期采用者，整个领域正在快速变化，我们的发现很可能暂时还无法推广到其他组织或环境（更多局限性见附录）。这项研究也反映了这种不确定性：这些发现是细腻的，没有出现单一共识或清晰指令。但它确实提出了一个问题：我们如何能够深思熟虑且有效地应对这些变化。

    为了跟进这项初步工作，我们正在采取若干步骤。我们正在与 Anthropic 的工程师、研究人员和领导层讨论，以应对这些发现所带来的机遇与挑战。这包括研究我们如何把团队聚在一起并相互协作、如何支持职业发展，以及如何为 AI 增强的工作建立最佳实践（例如以我们的 AI 流利度框架为指引）。我们也在把这项研究扩展到工程师之外，以理解 AI 转型如何影响组织内的各种角色，并支持诸如 CodePath 等外部组织在面向 AI 辅助未来时调整计算机科学课程。展望未来，我们也在考虑一些结构性做法——随着 AI 能力进步，这些做法可能会越来越相关，例如组织内部角色演化或再技能培养的新路径。

    随着我们的思考逐渐成熟，我们预计将在 2026 年分享更具体的计划。Anthropic 是一个负责任的职场转型实验室；我们不仅想研究 AI 如何改变工作，也想从我们自己开始，试验如何深思熟虑地应对这种转型。

    局限性

    我们的调查发现受若干方法论局限性影响。我们通过便利抽样与目的抽样相结合来选择受访者（以确保覆盖组织的广泛代表性）。我们把调查发布在多个内部 Slack 频道，获得了 68 份回复；同时我们也从组织架构图中选取了研究与产品职能下 20 个多样化团队，并直接给每个团队的 5–10 位成员发送消息（总外联 n=207），最终获得了 64 份回复（回复率 31%）。我们访谈了最先回复的 53 位受访者。这里很可能存在一定的选择偏差：对 Claude 特别投入或意见强烈（正面或负面）的人更可能回复，而体验更中性的员工可能代表不足。

    此外，回复也可能受到社会期望偏差的影响（因为回复并非匿名，且所有参与者都是 Anthropic 员工，受访者可能夸大对 Claude 影响的正面评价）以及近因/回忆偏差的影响（要求参与者回忆 12 个月前的生产力与使用模式，容易出现记忆失真）。另外，如前所述，生产力总体上很难估计，因此这些自述应当谨慎看待。这些自述感知应与我们更客观的 Claude Code 使用数据结合解读；未来研究也将受益于匿名数据收集与更稳健、经验证的测量工具。

    我们的 Claude Code 分析使用跨时间段的比例抽样，这意味着我们只能衡量任务分布的相对变化，而不能衡量工作量的绝对变化。例如，当我们报告“实现新功能”的占比从 14% 增至 37% 的 Claude Code 使用中，并不一定表示总的新功能工作量更多。

    最后，这项研究是在 2025 年 8 月进行的，当时 Claude Sonnet 4 与 Claude Opus 4 是我们最先进的模型。鉴于 AI 发展的快速步伐，我们观察到的模式可能已经随着更新模型的推出而发生变化。
